## Key Architectural Decisions

### Next.js App Router
- All pages under `src/app/` use the App Router (not Pages Router)
- `layout.tsx` files handle shared layouts (sidebar persistence)
- Server Components by default; add `"use client"` only for interactive components
- API routes under `src/app/api/` use Route Handlers

### Layout Nesting
```
src/app/layout.tsx                  вҶҗ Root layout (html, body, fonts)
  в””в”Җв”Җ src/app/talent/layout.tsx     вҶҗ Talent sidebar + content wrapper
        в””в”Җв”Җ src/app/talent/*/page.tsx вҶҗ Individual pages render inside content area
  в””в”Җв”Җ src/app/admin/layout.tsx      вҶҗ Admin sidebar + content wrapper
        в””в”Җв”Җ src/app/admin/*/page.tsx
```

The talent and admin layouts are separate components but share the same `SidebarLayout.tsx` structural wrapper. The sidebar content (nav items, user info) differs by role.

### Data Fetching Pattern
- **Server Components:** Fetch data directly via Prisma in page components (for initial page loads)
- **Client Components:** Use SWR or React Query for data that needs polling/revalidation (notification badges, AI chat)
- **API routes:** Used for mutations (POST/PUT/DELETE) and for client-side fetches

### State Management
- No Redux or Zustand needed вҖ” this is a data-display application, not a complex interactive app
- React Query / SWR for server state
- React useState/useReducer for local UI state (form inputs, modals, filters)
- Context for auth state (current user, role)
